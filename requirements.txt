The project is a lightweight mobile web application intended for collecting geographically anchored visual references while working in the field. The application opens directly in a mobile browser and displays an interactive map centered on the user’s current physical position. The primary goal is to allow the user to associate a small reference photograph with a precise geographic location so that the location can later be matched with a higher-quality production image or other spatial information.

The application operates entirely in the browser and stores collected data in a GitHub repository. Each captured reference image is uploaded as a file whose name encodes the geographic coordinates of the location where it was recorded. This design eliminates the need for a database or backend server and allows the repository itself to function as the data store.

This map may later grow into a more general field mapping utility rather than remaining limited to reference photographs. Future features may include: uploading other kinds of photos - not "reference" photos but the full-size production photos; creating osm features (Points or Polygons for new buildings) and uploading these to OpenStreetMap api. All these are just ideas for the future but I'd like to have an architecure flexible enough to allow for later implementation of these.

When the application loads, it first attempts to obtain the device’s geographic location using the browser geolocation capabilities provided by the Geolocation API. If the request succeeds, the application initializes the map centered on those coordinates with a zoom level of approximately 18 so that individual buildings are clearly visible. If the request fails or times out, the application does not initialize the map and instead displays a simple message indicating that location information is unavailable. Because the tool is intended for field use, the presence of a working location signal is considered a requirement for normal operation.

The map itself is rendered using MapLibre GL JS. The base visual layer of the map consists of satellite imagery so that the user can clearly recognize buildings and other physical structures from above. Over this imagery, a minimal overlay derived from OpenStreetMap data displays key cartographic features. These include roads, building outlines, major water bodies, and streams. The purpose of this overlay is to provide spatial orientation while keeping the map visually simple enough that the satellite imagery remains the primary visual reference.

The map includes the built-in geolocation control provided by MapLibre. This control displays the user’s current position as a moving dot and includes a standard “locate me” button that can recenter the map on the user’s location if they have moved away from the initial view. The application does not force the map to follow the user continuously, but the location marker updates as new GPS data arrives so that the user can see their movement relative to nearby buildings.

User interaction with the map is based on tapping a geographic location. When the user taps the map, the application records the coordinates of the tapped position and presents a small contextual action menu anchored to that location. This menu represents the available operations that can be performed at the selected geographic point. Initially the menu contains only one action, but it is designed to support additional actions in the future.

The current action offered in the menu, called "Create photo marker", is the creation of a photographic marker at the selected location.

When the user selects the photo marker action, the application launches the image acquisition workflow. A file input element is triggered so that the mobile browser presents its native interface allowing the user either to capture a new photograph with the device camera or select an existing image from the device gallery. The browser manages the interface for choosing between these sources, ensuring that the experience is consistent with the device’s native behavior.

Once the user selects or captures an image, the application processes the file locally in the browser. The original image produced by the device camera may be several megabytes in size, which would quickly cause the repository to grow too large. For this reason the application resizes and compresses the image before upload. The resizing process uses an off-screen canvas to scale the image down to a smaller resolution and compress it to a JPEG file that is typically below one hundred kilobytes. These images are intentionally low quality because they serve only as visual reference anchors rather than final production photographs.

During the image processing stage, the geographic coordinates from the map tap are formatted into a filename. Latitude and longitude values are rounded to a fixed precision so that filenames remain stable and consistent. The coordinates are combined into a string such as “latitude_longitude.jpg”. Because the coordinates are embedded directly in the filename, the repository itself becomes a spatial index: simply listing the files provides all recorded locations without the need for a separate metadata database.

The processed image file is uploaded to the repository using the GitHub REST API provided by GitHub. Each upload is implemented as a commit that adds a new file to the repository.

Authentication for the GitHub API is handled through a personal access token supplied by the user. The application provides a small interface element, represented by a key icon, that opens a popover dialog when clicked. This dialog contains an input field where the user can paste their GitHub token and a save button to store it. When saved, the token is written to the browser’s local storage so that it persists across sessions. Whenever the application performs an upload, the GitHub API module retrieves the stored token and includes it in the authorization header of the request. If the token is missing or invalid, the upload fails and the user receives a generic notification that the operation did not succeed.

After the upload request completes successfully, the application immediately adds a marker to the map at the recorded coordinates.

To keep the codebase maintainable and extensible, the application is divided into small independent JavaScript modules. Each module resides in its own file and exports a single primary function responsible for one clearly defined task. A location module handles the initial geolocation request and returns the coordinates required to initialize the map. A map module creates the MapLibre map instance, attaches the geolocation control, manages markers, and exposes a way to register map click events.

A separate interaction module receives coordinates from the map click event and displays the contextual action menu. This module returns the action selected by the user together with the original coordinates so that other modules can respond accordingly. The image acquisition module manages the file input element and returns the image selected or captured by the user. The image processing module performs resizing and compression using a canvas transformation and produces the small JPEG file that will be uploaded.

An authentication module manages the user interface for storing and retrieving the GitHub access token from local storage. The GitHub API module performs the actual upload operation by constructing and sending the commit request that writes the image file to the repository.

Finally, a coordination module connects these pieces together. It listens for map taps, invokes the interaction menu, launches the image capture process when requested, passes the resulting image through the resizing stage, constructs the filename from the coordinates, and calls the GitHub upload module. When the upload succeeds, it instructs the map module to create the corresponding marker.

However, it's worth considering an event-based system instead of the coordination module.

The resulting system is a compact field-mapping tool that allows geographic observations to be recorded quickly and with minimal infrastructure.
